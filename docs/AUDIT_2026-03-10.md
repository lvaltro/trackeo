## 1. Resumen ejecutivo

Trackeo.cl es una plataforma SaaS de rastreo GPS para pymes y personas, construida sobre un stack moderno (React 19 + Express + Supabase + Traccar), actualmente en **Fase 1 de consolidación con score 3/10** de enterprise-readiness.

El proyecto tiene una visión estratégica clara y documentación sólida, pero el código presenta una brecha importante entre la arquitectura *planeada* y la *real*. El backend fue recientemente modularizado (sprint 2026-03-05), lo que mejora la mantenibilidad, pero todavía faltan las capas de seguridad críticas: validación de inputs, RBAC, aislamiento multitenancy en tablas standalone, tests y logging estructurado.

**Fortalezas reales:**
- Separación entre `core/` (lógica pura) y `server/` (HTTP) bien ejecutada
- Documentación de reglas arquitectónicas (CLAUDE.md) existente y referenciada
- Sistema de geocoding con caché por proximidad y rate-limiting bien diseñado
- Middleware de ownership activo en rutas críticas

**Riesgos críticos:**
- Cero validación de inputs en todos los endpoints
- Sin RBAC implementado
- Tablas standalone (`maintenance`, `documents`, `notifications`, `weekly_stats`) sin `organization_id` → IDOR multitenancy
- Cero tests
- Sistema de autenticación acoplado a Traccar (IdP incorrecto para SaaS)
- Archivos `live-share` y geocoding cache en memoria → pérdida de estado en restart

**Score actual: 5.1/10** (funcional para uso individual; necesita hardening antes de escalar)

## 3. Arquitectura actual del backend

### Organización actual

El backend está organizado **por capas técnicas**, no por dominios de negocio:

```
server/
  routes/         ← routing HTTP
  middleware/     ← cross-cutting concerns
  lib/            ← utilidades compartidas

core/
  maintenance/    ← dominio mantenimiento
  documents/      ← dominio documentos
  notifications/  ← dominio notificaciones
  live-share/     ← feature compartir viaje
  geocoding/      ← servicio transversal
  jobs/           ← tareas programadas
```

Hay un **híbrido**: `server/` está por capas técnicas, y `core/` está parcialmente por dominio. La separación existe pero no es coherente. Las rutas no tienen un controlador intermediario — van directo a `core/`:

```
routes/maintenance.js → llama directamente a core/maintenance/index.js
```

Esto es correcto por simplicidad actual, pero a medida que los dominios crezcan, hará falta una capa de orquestación.

### Stack de middleware (orden de ejecución)

```
requestId → helmet → json parser → cors → apiLimiter → [route] → errorHandler
```

La cadena es correcta. El `errorHandler` está al final (bien). `cors` va antes del rate limiter (correcto para preflight OPTIONS).

### Patrones detectados

**Bien implementados:**
- `core/` es HTTP-agnostic — no importa Express en ningún módulo de dominio
- Todos los módulos en `core/` exportan funciones puras con `module.exports`
- `server/middleware/ownership.js` verifica propiedad vehicleId con caché de 60s

**Problemas detectados:**

| Problema | Archivo | Detalle |
|---|---|---|
| Sin capa de validación | todos los routes/ | `req.body` se pasa crudo a core/ sin esquema Zod |
| Sin controladores — routes llaman directamente a core/ | todos los routes/ | Acoplamiento route→domain directamente |
| `core/live-share/index.js` usa `new Map()` en memoria | `core/live-share/index.js` | Estado perdido en restart (viola R6) |
| `core/notifications/index.js` tiene credenciales Supabase en docstrings | `core/notifications/index.js` | Riesgo de exposición accidental |
| `server/lib/logger.js` usa `fs.appendFile` | `server/lib/logger.js` | Bloquea el event loop si el disco es lento |
| Sin AuditLog activo | todos los routes/ | El modelo Prisma existe pero nunca se escribe |
| `server/routes/notifications.js` no verifica vehicleId ownership | `server/routes/notifications.js` | Las notificaciones de cualquier usuario son accesibles |
| `server/routes/jobs.js` usa `requireAuth` pero no requiere rol admin | `server/routes/jobs.js` | Cualquier usuario autenticado puede disparar jobs |

### Estructura backend recomendada (sin modificar nada aún)

El análisis sugiere migrar hacia organización **por dominio** en el backend:

```
server/
  domains/
    vehicles/     routes + validation
    maintenance/  routes + validation
    documents/    routes + validation
    geofences/    routes + validation
    trips/        routes + validation
    notifications/ routes + validation
    analytics/    routes + validation
  shared/
    middleware/
    lib/
    jobs/
```

Esto alinea el backend con los dominios de negocio reales y facilita asignar ownership por módulo.

## 5. Dominios de negocio detectados

Basado en el análisis del código real (routes, core/, hooks, componentes, schema.prisma):

### Dominio 1: Auth & Identity
**Propósito**: Login, sesión, refresh, permisos, roles de usuario.
**Archivos relacionados**: `server/middleware/auth.js`, `src/screens/LoginScreen.jsx`, `prisma/schema.prisma` (User, Organization, Role, Permission), `docs/RBAC.md`
**Estado**: Parcialmente implementado. Auth via Traccar JSESSIONID funciona pero está acoplada al IdP equivocado. Supabase Auth preparado pero no integrado. RBAC en schema pero sin wiring.
**Límites**: Mezclado — auth Traccar y auth SaaS conviven sin separación clara.

### Dominio 2: Vehicles & Devices
**Propósito**: Gestión de la flota: vehículos, dispositivos GPS, asignación, estado.
**Archivos relacionados**: `src/hooks/useVehicleTracker.js`, `src/api/traccarApi.js`, Supabase tables `devices`, `vehicles`, `vehicles_status`, `prisma/schema.prisma` (Device, Vehicle, VehicleDriver)
**Estado**: Funcional para lectura. Escritura/gestión (crear vehículo, asignar dispositivo) no existe en la UI.
**Límites**: Confuso — `vehicle.id` en frontend = `device_id` de `vehicles_status`, no el UUID de la tabla `vehicles`. El join es correcto pero el naming es engañoso.

### Dominio 3: Live Tracking
**Propósito**: Posición en tiempo real, mapa, estado online/offline, ignición.
**Archivos relacionados**: `src/hooks/useVehicleTracker.js`, `src/components/MapView.jsx`, `src/screens/DashboardLayout.jsx`, `app/api/worker/process/route.ts`
**Estado**: Funcional vía Supabase Realtime. Worker en Vercel procesa posiciones desde Traccar → QStash → Supabase.
**Límites**: Pipeline Traccar→Worker no completamente implementado (stub en `app/api/traccar/route.ts`).

### Dominio 4: Trip History
**Propósito**: Historial de recorridos, polyline, estadísticas de viaje.
**Archivos relacionados**: `src/api/traccarApi.js` (positions, reports), `src/components/HistoryView.jsx`, Supabase table `device_positions`
**Estado**: Parcialmente implementado. Posiciones se almacenan pero historial de viajes como entidad (Trip) existe en schema sin endpoints para verlos.
**Límites**: `device_positions` (raw telemetría) y `Trip` (entidad procesada) están desconectados. No hay lógica de "fin de viaje".

### Dominio 5: Geofences
**Propósito**: Zonas geográficas, alertas de entrada/salida.
**Archivos relacionados**: `src/components/GeofencesView.jsx`, `prisma/schema.prisma` (Geofence, GeofenceEvent), `src/api/traccarApi.js`
**Estado**: UI parcial. Geofences se gestiona desde Traccar (no desde el SaaS propio). No hay endpoints Express para geofences.
**Límites**: Completamente delegado a Traccar. Sin lógica propia.

### Dominio 6: Maintenance
**Propósito**: Registro de mantenciones, alertas de km/fecha, historial.
**Archivos relacionados**: `core/maintenance/index.js`, `server/routes/maintenance.js`, `src/hooks/useMaintenance.js`, `src/api/maintenanceApi.js`, `src/components/MaintenanceDashboard.jsx`, Supabase table `maintenance_records`
**Estado**: Funcional. Completo: CRUD, dual-mode, ownership verificado.
**Límites**: Relativamente bien delimitado. Problema: `organization_id` ausente en tabla.

### Dominio 7: Documents
**Propósito**: Documentos del vehículo (seguro, revisión técnica, licencia), alertas de vencimiento.
**Archivos relacionados**: `core/documents/index.js`, `server/routes/documents.js`, `src/hooks/useDocuments.js`, `src/api/documentsApi.js`, Supabase table `vehicle_documents`, `core/jobs/docReminders.js`
**Estado**: Funcional. CRUD, expiration status, reminder job.
**Límites**: Bien delimitado. Mismo problema: sin `organization_id`.

### Dominio 8: Notifications
**Propósito**: Alertas del sistema, notificaciones de usuario, estado leído/no leído.
**Archivos relacionados**: `core/notifications/index.js`, `server/routes/notifications.js`, `src/hooks/useNotifications.js`, `src/api/notificationApi.js`, Supabase table `notifications`
**Estado**: Funcional. Sin filtro por org — cualquier usuario autenticado ve sus notificaciones pero no hay isolación multitenancy.
**Límites**: Sin `organization_id`, sin categorización, sin prioridad.

### Dominio 9: Analytics & Scoring
**Propósito**: Estadísticas semanales, score de conductor, km recorridos.
**Archivos relacionados**: `core/jobs/weeklyStats.js`, `server/routes/weeklyStats.js`, `src/hooks/useWeeklyStats.js`, `src/api/weeklyStatsApi.js`, `src/components/WeeklyUsageWidget.jsx`, Supabase table `vehicle_weekly_stats`

### Dominio 10: Live Share (Viaje Seguro)
**Propósito**: Compartir ubicación en tiempo real con link público temporal.

**Límites**: Feature fantasma — existe en el schema y en el frontend pero sin backend real.

### Dominios definidos en schema pero sin implementación
- **Billing / Subscriptions**: modelos `Plan`, `Subscription`, `Payment` en schema.prisma, sin endpoints ni UI
- **School transport / Fleet operations**: mencionado en visión estratégica, sin código

## 6. Estructura de datos y entidades

### Arquitectura dual de base de datos

El proyecto tiene **dos capas de datos completamente distintas** que coexisten en el mismo Supabase:

```
┌─────────────────────────────────────────────────────────┐
│  CAPA 1: Prisma + PostgreSQL (lógica SaaS)              │
│  Managed by: prisma/schema.prisma                       │
│  Usado para: Orgs, Usuarios, Planes, Billing            │
│  Estado: Schema completo, migraciones pendientes prod   │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  CAPA 2: Supabase directo (pipeline IoT)                │
│  Managed by: standalone SQL migrations                   │
│  Usado para: devices, vehicles, positions, events       │
│  Estado: Activo en producción, Realtime funcionando     │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  CAPA 3: Supabase directo (datos de negocio)            │
│  Managed by: standalone SQL migrations                   │
│  Usado para: maintenance, documents, notifications      │
│  Estado: Activo, pero SIN organization_id               │
└─────────────────────────────────────────────────────────┘
```

### Entidades principales y relaciones

```
Organization ──< User
Organization ──< Vehicle
Vehicle >── Device (IMEI)
Device ──< vehicles_status (1:1 estado actual)
Device ──< device_positions (time-series)
Device ──< device_events (alertas)
Vehicle ──< maintenance_records (SIN org_id)
Vehicle ──< vehicle_documents (SIN org_id)
Vehicle ──< vehicle_weekly_stats (SIN org_id)
User ──< notifications (SIN org_id)
```

### Inconsistencias críticas detectadas

| Inconsistencia | Detalle | Riesgo |
|---|---|---|
| `vehicle.id` vs `device_id` | En `vehicles_status`, la PK es `device_id`. En frontend, `activeVehicle.id` = `device_id`, no el UUID de `vehicles`. Naming confuso. | Bugs silenciosos |
| `organization_id` ausente en 4 tablas standalone | `maintenance_records`, `vehicle_documents`, `notifications`, `vehicle_weekly_stats` no tienen `organization_id` | IDOR multitenancy crítico |
| Capa 1 (Prisma) y Capa 3 (standalone) tienen modelos duplicados | Prisma tiene `MaintenanceRecord`, `VehicleDocument`, `Notification` — y también existen como tablas standalone | Sincronización confusa |
| `device_positions` sin particionamiento por tiempo verificado | Time-series sin partición por mes → performance degradará al crecer | Escalabilidad |
| `AuditLog` en schema Prisma nunca escrito | Existe el modelo pero `server/` nunca llama a `prisma.auditLog.create()` | Sin trazabilidad |
| `Trip` en schema Prisma sin endpoints ni pipeline | Existe en schema pero no hay lógica para detectar inicio/fin de viaje | Feature incompleta |

### Evaluación de escalabilidad

El esquema actual es adecuado para **< 1.000 dispositivos activos**. Para escalar a verticales enterprise se necesitará:

1. Particionamiento de `device_positions` por tiempo
2. `organization_id` en todas las tablas
3. Índices en `device_id + recorded_at` en `device_positions`
4. Separar `vehicles_status` (hot path) de `device_positions` (cold path, ya separado ✅)
5. Modelo de `Trip` procesado — calcular desde posiciones raw no es viable a escala

## 7. Flujo de integraciones y telemetría

### Pipeline actual GPS → Frontend

```
Dispositivo GPS (Micodus/Teltonika)
    ↓ protocolo propietario
Traccar Server (api.trackeo.cl:8082)
    ↓ webhook HTTP POST
[STUB] app/api/traccar/route.ts (Next.js en Vercel)
    ↓ enqueue
Upstash QStash (cola de mensajes)
    ↓ delivery garantizado
[STUB] app/api/worker/process/route.ts (Next.js en Vercel)
    ↓ insert
Supabase PostgreSQL
    ├── device_positions (histórico)
    ├── device_events (alertas)
    └── vehicles_status (estado actual)
         ↓ Supabase Realtime (WebSocket)
Frontend useVehicleTracker.js
    ↓ render
MapView.jsx (Leaflet)
```

### Problemas críticos en el pipeline

| Problema | Detalle | Impacto |
|---|---|---|
| Worker es un STUB | `app/api/worker/process/route.ts` no está completamente implementado | Pipeline roto o incompleto |
| Ingestor es un STUB | `app/api/traccar/route.ts` no está completamente implementado | Sin rate limiting en ingesta |
| Sin normalización de protocolo | Distintos dispositivos (Micodus, Teltonika) envían formatos distintos | Cuando se agregue otro proveedor, hay que reescribir |
| Sin dead-letter queue | Si el worker falla, la posición se pierde | Pérdida de datos |
| Sin validación de IMEI en worker | Worker no verifica que el IMEI exista en `devices` | Datos huérfanos |
| Geocoding acoplado al backend Express | Reverse geocoding solo disponible via servidor Express, no en worker | Dependencia arquitectónica |

## 8. Preparación para eventos y automatizaciones

### Estado actual del patrón event-driven

El proyecto **no tiene un sistema de eventos explícito**. Los "eventos" actuales son implícitos:

| Evento lógico | Cómo se maneja hoy | Problema |
|---|---|---|
| Posición actualizada | Worker escribe en Supabase → Realtime notifica frontend | No se emite evento a otros consumidores |
| Dispositivo offline | No detectado activamente | Sin alert real |
| Mantenimiento vencido | `core/jobs/docReminders.js` via `setInterval` | Polling, no event-driven |
| Documento vencido | `core/jobs/docReminders.js` | Polling |
| Viaje terminado | No existe | Sin detección de trips |
| Alerta de velocidad | `device_events` existe en DB | Sin consumidor en frontend |
| Geofence violada | `device_events` existe en DB | Sin lógica de alerta |
| Onboarding completado | No existe | Sin trigger |

### Preparación para n8n / automatizaciones externas

**Actualmente: 0% preparado**. No existe:
- Webhook outbound para eventos
- Event bus o message queue interno
- Tabla de `events` o `outbox`
- API pública para suscribirse a eventos

### Recomendación de arquitectura de eventos

Para conectar con n8n u otras automatizaciones:

```
core/events/
  eventEmitter.js     ← Node.js EventEmitter o simple Supabase insert
  types.js            ← Enum de tipos: POSITION_UPDATED, DEVICE_OFFLINE, etc.

Supabase tabla: domain_events
  id, org_id, entity_type, entity_id, event_type, payload, created_at

Webhook outbound (cuando org tiene configurado):
```

## 10. Riesgos y deuda técnica

### 🔴 CRÍTICO — Bloquea producción segura

| # | Riesgo | Detalle | Impacto |
|---|---|---|---|
| C1 | **Sin org_id en tablas standalone** | `maintenance_records`, `vehicle_documents`, `notifications`, `vehicle_weekly_stats` sin `organization_id`. Un usuario puede ver/modificar datos de otro org si adivina el `vehicleId`. | IDOR multitenancy — fuga de datos entre clientes |
| C2 | **Sin validación de inputs** | Todos los endpoints aceptan `req.body` crudo. Sin sanitización, tipo-checking ni límites. | Injection, crashes, datos corruptos |
| C3 | **Sin RBAC** | Cualquier usuario autenticado puede llamar a cualquier endpoint (excepto los protegidos por ownership). No hay roles granulares. | Escalada de privilegios |
| C4 | **Worker en Vercel es un stub** | El pipeline completo Traccar → Supabase no está terminado. Si el worker falla silenciosamente, se pierde telemetría. | Pérdida de datos GPS |
| C5 | **Auth acoplada a Traccar** | Traccar es el IdP para el SaaS. Si Traccar cae o cambia su API, el login entero falla. | Single point of failure en autenticación |

### 🟠 ALTO — Impacta calidad y mantenibilidad

| # | Riesgo | Detalle |
|---|---|---|
| A1 | **Cero tests** | Sin unit tests, sin integration tests, sin E2E. Cualquier refactor o feature nueva tiene riesgo alto de regresión. |
| A2 | **live-share en memoria** | Los shares se pierden con cada restart de PM2. En producción con clientes activos, esto es un bug. |
| A3 | **logger.js con fs.appendFile** | Operación síncrona de I/O en el event loop. Bajo carga puede degradar performance. |
| A4 | **Sin AuditLog** | El modelo Prisma existe pero nunca se escribe. Sin trazabilidad. Requerido para enterprise. |
| A5 | **MaintenanceDashboard.jsx de 70KB** | Componente monolítico. Imposible de testear, difícil de mantener. |
| A6 | **OnboardingWizard.jsx de 42KB** | Feature compleja sin backend real. |
| A7 | **hooks stub sin backend** | `useEngineControl.js`, `useProtection.js`, `useTraccarSocket.js` existen en producción sin pares en el backend. |

### 🟡 MEDIO — Deuda técnica acumulable

| # | Riesgo | Detalle |
|---|---|---|
| M1 | **Sin CI/CD** | Deploy manual. Un error afecta producción directamente sin gate de calidad. |
| M2 | **Sin API versioning** | Cuando haya consumidores externos (mobile, instaladores), los cambios de API romperán clientes. |
| M3 | **Logging no estructurado** | `logError()` custom en vez de `pino`. Difícil de parsear en herramientas de observabilidad. |
| M4 | **Feature flags no conectados** | Schema existe, código no los usa. |
| M5 | **landing/ mezclada en el repo** | Proyecto Next.js completamente independiente. Complica el deploy. |
| M6 | **app/ (worker) mezclado en el repo** | Otro proyecto independiente. Debería ser workspace claramente demarcado. |
| M7 | **Naming confuso vehicle.id vs device_id** | Fuente de bugs al agregar nuevas features que trabajan con vehículos. |

### Qué va en CLAUDE.md (raíz) vs ai/

**CLAUDE.md raíz** (siempre cargado, máximo 150 líneas):
- Reglas arquitectónicas R1-R11 (inmutables)
- Stack actual (una línea por capa)
- Fase actual del proyecto (una línea)
- Paths críticos (5-8 paths)
- Referencia a `ai/` para contexto detallado

**ai/core/** (bajo demanda, cargado por tipo de tarea):
- Decisiones técnicas detalladas
- Modelo de datos completo
- Flujos de autenticación
- Convenciones de código

**ai/domains/** (cargado solo cuando se trabaja en ese dominio):
- Un archivo por dominio
- Archivos relevantes, patrones específicos, gotchas, decisiones pasadas

**Por qué esta estructura reduce token waste:**
- En vez de cargar todo CLAUDE.md para una tarea simple, se carga solo el contexto del dominio relevante
- Los `playbooks` eliminan instrucciones repetitivas en cada sesión
- El archivo `CONTEXT.md` (estado actual) se actualiza frecuentemente sin contaminar las reglas estables
